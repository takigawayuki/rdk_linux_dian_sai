### 方向判断
- deta_x ：
  - 正值 ：目标在基准点的 右侧
  - 负值 ：目标在基准点的 左侧
  - 0 ：目标在基准点的水平中心线上
- deta_y ：
  - 正值 ：目标在基准点的 下方
  - 负值 ：目标在基准点的 上方
  - 0 ：目标在基准点的垂直中心线上

### 偏差程度
- 绝对值越大，偏差越大
- 绝对值越小，目标越接近基准点
- 当 deta_x 和 deta_y 都接近 0 时，目标位于激光点上

✅ dx（左右控制）
dx = center_x - base_x
情况	含义	dx 符号
目标在右边	center_x > base_x	dx > 0
目标在左边	center_x < base_x	dx < 0
👉 所以：
dx > 0 → 云台往右转
dx < 0 → 云台往左转

✅ dy（上下控制）
dy = base_y - center_y
注意这里反了一次
情况	含义	dy 符号
目标在上面	center_y < base_y	dy > 0
目标在下面	center_y > base_y	dy < 0
👉 所以：
dy > 0 → 云台往上抬
dy < 0 → 云台往下压


你现在打印的数据解释
dx = -0.179
dy = -0.144
说明：
dx < 0 → 目标在左边
dy < 0 → 目标在下方
👉 目标在画面 “左下角”
👉 云台动作应该是：
    向左转
    向下压


dx 和 dy 的正负是根据云台的控制逻辑来的，跟 OpenCV 坐标系本身没关系，是人为定义的。
拆开来看：
dx = -(BASE_POINT[0] - center[0]) * 0.001
BASE_POINT[0] - center[0] = 目标在基准点左边时为正，右边时为负
加了负号，所以目标在右边时 dx 为正 → 云台向右转
dy = +(BASE_POINT[1] - center[1]) * 0.001
BASE_POINT[1] - center[1] = 目标在基准点上方时为正（因为图像 y 轴向下），下方时为负
没加负号，所以目标在上方时 dy 为正 → 云台向上转
简单说就是：正负号是用来匹配云台电机方向的，如果云台转反了就改符号。

其实 dx/dy 本来就是在 OpenCV 坐标系下算的，只是 dy 做了一次语义翻转：
图像 y 轴向下，但人说"目标在上方"时希望云台往上抬，所以用 base_y - center_y 
而不是 center_y - base_y，
让正值对应"往上"这个直觉方向。这是故意的，不是 bug。

左上角 (0,0)，x 向右，y 向下
(0,0) ─────────→ x+
  │
  │
  ↓

单位是像素

激光笔坐标系
和相机坐标系平行，但原点偏移了一个物理距离（就是你说的视差问题）
你现在代码里没有处理这个变换，直接用图像中心当激光点，所以有偏差

云台坐标系
以云台旋转中心为原点
dx/dy 就是在这个坐标系下的控制量
你代码里的正负号就是在做"图像坐标系 → 云台坐标系"的方向对齐
按照道理来说云台的水平控制定义为：正值（Yaw）表示向右，负值表示向左
            云台的垂直控制定义为：正值（Pitch）表示向上，负值表示向下

你现在代码里实际只做了一个变换：
图像坐标系 → 云台控制量
dx = -(BASE_POINT[0] - center[0]) * 0.001
dy = +(BASE_POINT[1] - center[1]) * 0.001
